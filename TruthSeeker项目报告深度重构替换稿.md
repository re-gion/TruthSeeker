## 1.3.2 多智能体阶段式协同鉴伪与跨模态交叉验证

【修改说明】

原节标题建议由“多智能体交叉验证系统”调整为“多智能体阶段式协同鉴伪与跨模态交叉验证”。重点不再放在抽象的多 Agent 辩论，而是突出当前代码中的阶段式拓扑、逻辑质询回退、证据板共享和最终裁决机制。

【建议替换正文】

TruthSeeker 的第二项核心特色是面向恶意 AIGC 鉴伪任务构建了阶段式多智能体协同机制。系统将复杂研判任务拆解为检材侧鉴伪、外部情报溯源、逻辑质询和综合裁决四个职责边界清晰的子过程，并由 Forensics、OSINT、Challenger、Commander 四类 Agent 分别承担。整体流程遵循 `Forensics -> Challenger -> OSINT -> Challenger -> Commander -> END` 的阶段式拓扑：Forensics Agent 先围绕上传检材进行跨模态取证和底层检测工具调用，Challenger Agent 对其证据充分性与推理闭环进行质询；若检材侧证据满足阶段要求，任务进入 OSINT Agent，由其补充 URL、域名、公开来源、社会工程风险和相似案例线索；随后 Challenger Agent 再次检查溯源证据是否存在缺口，最后由 Commander Agent 汇总证据、分歧、残留风险和人机协同摘要，形成最终裁决。

该机制的优势在于把多源检测结果转化为可追踪的证据闭环。Fed-MBPR 提供的底层联邦多模态模型能力被封装为 Agent 可调用的检测工具，Forensics Agent 可以调用图像、音频、视频、文本等模态相关的鉴伪能力，OSINT Agent 可以调用恶意 URL、外部情报和公开来源检索能力。Agent 之间不直接以单次模型输出作为结论，而是把工具结果、模型判断、证据来源、置信度、质询意见和残留风险写入共享状态，由 Challenger Agent 对每个阶段进行二次审查。只有当当前阶段达到满意阈值，或达到轮次上限并记录残留风险，系统才会进入下一阶段。

因此，TruthSeeker 的“交叉验证”不是简单的多个检测器取平均，而是围绕证据链条进行的任务分解、工具调用、逻辑质询和裁决融合。它能够在图片、文本、音频、视频和外部情报之间建立相互校验关系，既降低单一模型误判带来的风险，也让最终报告能够解释“系统为什么得出该结论、哪些证据支撑该结论、哪些问题仍然存在不确定性”。

## 1.3.3 面向分歧消解的人机协同研判机制

【修改说明】

原节建议改写为“面向分歧消解的人机协同研判机制”。正文应弱化“外部专家必然介入”的叙述，改为用户主导、系统提示、必要时邀请外部专家补充意见的协同流程。

【建议替换正文】

TruthSeeker 的第三项核心特色是将用户判断纳入 Agent 证据闭环，形成面向分歧消解的人机协同研判机制。系统在自动检测过程中持续记录每轮 Agent 的质询意见、置信度变化、证据缺口和可疑点。当 Challenger Agent 发现同一阶段连续多轮低置信、置信度变化停滞或关键证据无法闭合时，系统不会直接给出武断结论，而是生成结构化协同上下文，将当前分歧点、已采集证据、待确认样本、疑似风险和需要用户判断的问题推送给用户。

在协同阶段，用户可以根据自身掌握的业务背景、样本来源和现场经验，对系统提出的问题进行确认、纠正或补充。若用户无法独立判断，也可以通过系统生成的邀请入口让外部专家参与讨论。外部专家的作用是补充专业意见，而不是替代用户或系统完成最终裁决。协同结束后，Commander Agent 会对人类反馈进行摘要，用户可以编辑并确认摘要内容；确认后的摘要会回流到后续 Agent 推理和最终报告中，作为解释结论、记录残留风险和生成处置建议的重要依据。

该机制解决了恶意 AIGC 鉴伪中常见的“机器证据不足但人工背景关键”的问题。例如，系统可能识别到图片存在生成痕迹，但无法判断其是否经过合法二次编辑；也可能发现 URL 存在威胁情报命中，但需要用户确认其传播场景。通过人机协同，TruthSeeker 能够把机器检测、公开情报、用户背景知识和外部专业意见纳入同一证据链条，并在任务结束后将有价值的协同经验沉淀为个人经验库内容，为后续相似任务提供检索增强。

## 2.1.2 多智能体阶段式协同鉴伪系统

【修改说明】

本节应替换原“多智能体交叉验证系统”的旧叙述。建议明确当前系统的阶段式拓扑、共享状态、四类 Agent 职责、Challenger 放行规则、人机协同触发条件、Commander 权重融合公式和 Fed-MBPR 工具底座关系。

【建议替换正文】

TruthSeeker 的多智能体协同鉴伪系统以 LangGraph 状态图为运行骨架，以共享状态对象为证据与推理载体，构建了从检材侧鉴伪到外部溯源再到综合裁决的阶段式分析链路。系统的核心拓扑为：

$$
A_f \rightarrow A_q \rightarrow A_o \rightarrow A_q \rightarrow A_m \rightarrow END
$$

其中，$A_f$ 表示 Forensics Agent，负责检材侧鉴伪；$A_o$ 表示 OSINT Agent，负责外部情报溯源；$A_q$ 表示 Challenger Agent，负责逻辑质询和阶段放行判断；$A_m$ 表示 Commander Agent，负责综合裁决和报告生成。该拓扑避免了多个 Agent 同时输出互不约束结论的问题，使每一阶段都必须经过质询与证据充分性检查。

系统运行时维护统一的任务状态，包括上传检材、检材类型、分析阶段、阶段轮次、证据文件、工具调用结果、Agent 日志、置信度历史、质询记录、残留风险、人机协同状态、溯源图谱和最终裁决等字段。每个 Agent 不直接覆盖其他 Agent 的判断，而是在共享证据板上补充结构化结果。证据板可抽象表示为：

$$
B_t = \{E_t, T_t, C_t, R_t, H_t\}
$$

其中，$E_t$ 表示第 $t$ 时刻已经积累的检材证据和外部来源，$T_t$ 表示工具调用结果，$C_t$ 表示 Agent 置信度与裁决倾向，$R_t$ 表示 Challenger 发现的问题和残留风险，$H_t$ 表示人机协同反馈。最终裁决不是某个单点模型输出，而是上述状态随阶段推进不断累积后的综合结果。

Forensics Agent 面向检材本身开展鉴伪分析。系统可根据输入类型调用图片、音频、视频、文本等检测工具，并将 Fed-MBPR 底座提供的底层模型能力封装为可调用工具，用于识别 AIGC 生成痕迹、深度伪造风险、文本生成概率、文件哈希风险和样本元信息。Forensics Agent 的输出包括模态级风险判断、AIGC 概率、关键证据、工具置信度和需要后续核验的问题。

OSINT Agent 面向外部情报和传播链路开展溯源分析。其任务包括对 URL、域名、公开网页、相似案例、社会工程风险和威胁情报线索进行查询与整理。OSINT Agent 不仅判断样本是否可疑，还需要说明可疑性的来源，例如外部来源命中、公开资料矛盾、传播场景异常或与既有案例存在相似模式。公开案例库 RAG 和个人经验库 RAG 在该阶段提供参考材料，但其作用是辅助分析和补充证据线索，而不是直接替代当前任务证据。

Challenger Agent 是系统证据质量控制的核心。每当 Forensics 或 OSINT 完成阶段输出后，Challenger Agent 会检查当前证据是否存在高风险缺口、推理跳跃、来源不足、置信度过低或结论与证据不匹配等问题。系统将问题按严重程度计入质量评分，可用如下方式表达：

$$
q_t = \max(q_{floor}, \min(q_{model}, 1 - 0.30n_h - 0.15n_m - 0.05n_l))
$$

其中，$q_t$ 表示第 $t$ 轮阶段质量评分，$q_{model}$ 表示模型和规则检查给出的基础评分，$n_h$、$n_m$、$n_l$ 分别表示高、中、低严重度问题数量，$q_{floor}$ 表示由强证据触发的置信度下限。该设计能够避免强证据被轻微文本问题过度拉低，同时也能阻止证据缺口较大的阶段被过早放行。

阶段放行规则可表达为：

$$
Release_t =
[(q_t \ge 0.8) \land (n_h = 0) \land \neg b_t]
\lor (k_t = 5)
\lor h_t
$$

其中，$0.8$ 为 Challenger 满意阈值，$b_t$ 表示模型或规则仍要求补充证据，$k_t$ 表示当前阶段轮次，$5$ 为阶段最大轮数，$h_t$ 表示人机协同后允许放行。若当前阶段未满足放行条件，Challenger Agent 会把任务路由回目标 Agent 进行补证；若达到轮次上限仍未完全解决，系统会把低置信点和未解决问题写入残留风险，而不是静默忽略。

当同一目标阶段连续三轮处于低置信状态，且相邻置信度变化小于 $0.08$，系统认为自动推理进入停滞状态，可触发人机协同：

$$
HumanTrigger_t =
\mathbb{I}\left(
\forall i \in \{t-2,t-1,t\}: q_i < 0.8
\land |\Delta q_i| < 0.08
\land target_i = target_t
\land k_t < 5
\right)
$$

在人机协同触发前，系统会优先检索个人经验库。如果存在与当前分歧相似的历史处置经验，Challenger Agent 可先将经验注入目标阶段提示并要求 Agent 重新分析；若经验不足以打破僵局，系统再进入人机协同流程。

Commander Agent 负责最终裁决与报告生成。系统默认采用 Forensics、OSINT、Challenger 三类证据权重进行融合，其中 Forensics 默认权重为 $0.45$，OSINT 默认权重为 $0.30$，Challenger 默认权重为 $0.25$。在工具降级或证据不足时，相应权重会被调低，残留风险会被显式写入最终报告。综合风险和置信度可概括为：

$$
S_{aigc} = 0.45p_f + 0.30r_o
$$

$$
Q = 0.45q_f + 0.30q_o + 0.25q_q
$$

$$
R = \max(S_{aigc}, r_o)
$$

其中，$p_f$ 表示检材侧 AIGC 风险，$r_o$ 表示外部情报风险，$q_f$、$q_o$、$q_q$ 分别表示取证、溯源和质询阶段置信度，$S_{aigc}$ 表示 AIGC 综合风险，$Q$ 表示综合置信度，$R$ 表示最终风险评分。系统根据上述结果及证据质量生成“伪造”“可疑”“真实”或“无法确定”等裁决类型，并同步生成关键证据、残留风险、处置建议和可追溯报告。

通过这种阶段式拓扑，TruthSeeker 将 Fed-MBPR 底层检测能力、多模态工具调用、外部情报检索、逻辑质询和最终裁决组织为一个闭环系统。其核心价值不只在于提高检测覆盖面，更在于让每一次结论都有明确的证据来源、质询过程、置信变化和人工反馈记录。

## 2.1.3 面向分歧消解的人机协同研判机制

【修改说明】

本节应替换原有以外部人员介入为主的叙述。当前系统更准确的表达是：当自动 Agent 推理出现低置信停滞或证据缺口时，系统将结构化分歧推送给用户，由用户确认、纠正、补充或邀请外部专家协助，最终由 Commander 形成摘要并回流到 Agent 状态。

【建议替换正文】

恶意 AIGC 鉴伪任务往往存在仅靠自动检测难以闭合的场景。例如，图片可能存在明显生成痕迹，但其是否属于恶意伪造还需要结合传播背景；URL 可能命中威胁情报，但是否与当前样本直接相关仍需用户确认；公开来源可能互相矛盾，需要熟悉事件背景的人补充上下文。为处理这类问题，TruthSeeker 在多智能体链路中设计了人机协同研判机制。

系统不会在每个任务中强制进入人工流程，而是在 Challenger Agent 检测到自动推理停滞时触发协同。触发条件包括：同一目标阶段连续三轮低于满意阈值、相邻置信度变化小于 $0.08$、问题集中在同一 Agent 或同一阶段，且尚未达到阶段最大轮次。触发后，系统会生成结构化协同上下文，包括当前阶段、目标 Agent、已确认的证据、仍存在的证据缺口、可疑样本链接、置信度变化、需要用户回答的问题和建议补充的材料。

协同流程可分为五步：

第一步，系统整理分歧。Challenger Agent 将自动推理中无法解决的问题转化为 `help_needed` 条目，每个条目包含问题类型、严重程度、目标 Agent、需要确认的内容和期望输出。这样用户看到的不是冗长日志，而是可以直接判断的研判问题。

第二步，系统优先检索个人经验库。若历史任务中存在相似问题、相似模态或相似处置方式，系统会先把经验内容作为检索增强材料回流给目标 Agent，尝试由自动流程完成补证。只有当经验不足以消除分歧时，系统才进入正式人机协同。

第三步，用户参与研判。用户可以在协同面板中查看系统整理的证据、分歧和问题，提交补充说明、确认某条证据是否可信、指出样本来源、纠正系统误判，或说明为什么某个风险在当前业务场景下并不成立。当用户需要外部专业意见时，可以生成邀请入口，让外部专家参与讨论并提交意见。

第四步，Commander Agent 生成协同摘要。协同结束后，Commander Agent 会把用户和外部参与者的有效反馈整理为结构化摘要，包括已确认事实、关键意见、仍未解决的问题、对后续判断的影响和建议处置方式。用户可以编辑并确认该摘要，避免系统把不准确的人类反馈直接写入最终推理链。

第五步，协同结果回流。经用户确认的摘要会进入共享状态，影响后续 Agent 复核和 Commander 最终裁决。该过程可抽象为：

$$
K_{t+1} = K_t \cup U_h \cup E_h \cup S_h
$$

其中，$K_t$ 表示当前任务知识状态，$U_h$ 表示用户补充的背景信息，$E_h$ 表示外部参与者补充的专业意见，$S_h$ 表示 Commander 生成并由用户确认的协同摘要。系统最终报告会展示协同结论及其对裁决的影响，使人工判断成为可审计、可追溯、可复用的证据组成部分。

人机协同结束后，系统还可以基于协同过程生成个人经验草稿。用户确认后，该经验会进入个人经验库，并按目标 Agent 建立检索索引。后续遇到相似样本或相似分歧时，系统能够优先检索历史经验，减少重复人工介入，提高复杂任务的处理效率。

## 2.1.4 跨模态数据融合与溯源图谱

【修改说明】

本节应补充当前系统已经实现的溯源图谱、引用来源、模型推断标记和质量指标，不应只停留在抽象的跨模态融合描述。

【建议替换正文】

TruthSeeker 的跨模态融合不是把图片、文本、音频、视频和外部情报简单拼接，而是将不同来源的证据统一映射到可追踪的溯源图谱中。系统在任务执行过程中记录上传检材、工具结果、公开来源、RAG 命中、Agent 判断、Challenger 质询、最终裁决等信息，并构建由节点、边和引用组成的证据图。

图谱节点包括 `artifact`、`entity`、`source`、`evidence`、`finding`、`claim`、`event`、`agent`、`verdict` 等类型。其中，`artifact` 表示上传检材，`source` 表示公开网页或外部情报来源，`finding` 表示工具检测结果，`claim` 表示模型或外部来源中的主张，`agent` 表示参与研判的智能体，`verdict` 表示最终裁决。图谱边包括 `extracted_from`、`derived_from`、`supports`、`refutes`、`contradicts`、`reviewed_by` 等关系，用于表达某条证据从哪里来、支持或反驳了什么结论，以及经过哪个 Agent 审查。

为避免模型推断与外部事实混淆，系统为节点和边记录引用信息。上传文件、工具调用结果、公开来源、公开案例 RAG 和个人经验 RAG 都会形成可追踪引用；缺少外部引用的模型综合判断会被标记为模型推断。前端溯源图谱可展示完整性、引用覆盖率、模型推断占比等质量指标，帮助用户理解哪些结论有明确来源支撑，哪些内容主要来自模型综合推理。

在跨模态场景中，该图谱能够把检材侧鉴伪证据与外部传播证据连接起来。例如，图片检测工具发现生成痕迹，OSINT Agent 又发现相关 URL 存在异常传播记录，公开案例库中存在相似攻击样式，Challenger Agent 则记录仍需确认的证据缺口。最终报告可以基于同一图谱说明各证据之间的支持、矛盾和不确定性，提升鉴伪结论的解释性和可复核性。

## 4.1 阶段式多智能体交叉验证与证据闭环创新

【修改说明】

原创新点标题建议由“多智能体交叉验证机制”调整为“阶段式多智能体交叉验证与证据闭环创新”。正文要突出阶段拓扑、质询回退和可解释证据链。

【建议替换正文】

TruthSeeker 的第一项系统创新在于将多智能体协同从“并列输出结论”改造为“阶段式证据闭环”。传统多模型检测系统往往把不同检测器结果进行简单投票或加权平均，一旦某个检测器误判，系统难以解释误差来源，也难以判断应补充哪类证据。TruthSeeker 将任务拆解为检材侧鉴伪、外部情报溯源、逻辑质询和综合裁决四个阶段，由 Forensics、OSINT、Challenger、Commander 四类 Agent 分工完成，并通过 Challenger Agent 在关键阶段执行质量门控。

该机制的创新点主要体现在三个方面。第一，系统把 Fed-MBPR 底层模型能力、第三方检测工具、公开情报和 RAG 检索结果统一纳入 Agent 可调用工具链，使跨模态检测能力能够服务于同一研判流程。第二，系统引入 Challenger Agent 作为独立质询环节，对每个阶段的证据充分性、推理一致性和置信度变化进行审查；当证据不足时，任务会回退到目标 Agent 继续补证，而不是直接进入最终裁决。第三，系统把工具结果、Agent 判断、质询记录、残留风险和最终结论写入同一证据链路，使报告能够展示结论形成过程。

与单一模型判别相比，该机制能够更好地适应恶意 AIGC 的跨模态特点。恶意内容可能同时包含生成图片、伪造文本、异常链接和可疑传播轨迹，任何单一模态都可能只能提供局部证据。TruthSeeker 通过阶段式交叉验证，将不同模态、不同来源和不同 Agent 的判断组织为可追溯闭环，从而提升系统在复杂样本上的解释性、稳健性和可复核性。

## 4.3 面向分歧消解的人机协同研判创新

【修改说明】

原创新点建议改名为“面向分歧消解的人机协同研判创新”。重点不再是外部人员参与本身，而是系统如何发现分歧、组织问题、吸收人工反馈并沉淀经验。

【建议替换正文】

TruthSeeker 的第三项创新在于把人类反馈设计为多智能体推理链路中的可控、可审计、可复用环节。传统自动鉴伪系统通常在模型置信度不足时直接给出“不确定”结论，或把人工判断作为系统外部的线下补充，导致人工意见难以进入后续推理，也难以在报告中追溯。TruthSeeker 则通过 Challenger Agent 主动识别低置信停滞和证据缺口，将复杂日志整理为用户可判断的结构化问题，并在必要时启动人机协同研判。

该机制首先解决了“何时需要人工”的问题。系统并不对所有任务都引入人工，而是在连续低置信、置信度变化停滞、同一阶段问题反复出现时触发协同，从而避免无谓打断自动流程。其次，它解决了“人工应判断什么”的问题。系统向用户展示的是当前分歧点、待确认事实、证据缺口和建议动作，而不是完整的底层日志。再次，它解决了“人工意见如何回到系统”的问题。Commander Agent 会把协同过程整理为摘要，用户确认后再写入任务状态，影响后续复核与最终报告。

更重要的是，系统把一次性人工反馈转化为可复用经验。协同结束后，用户可以将有效处置方法确认入库，后续相似任务触发低置信停滞时，系统会优先检索个人经验库并反馈给目标 Agent。由此，TruthSeeker 不仅在单次任务中结合了机器检测和人工判断，也在长期使用中形成面向用户场景的经验积累机制。

## 5.3 技术展望与未来优化

【修改说明】

原节中若仍将公开案例库、个人经验库、RAG 检索或 Agent 记忆写成未来工作，建议删除或改写。未来展望应放在更高层的模型服务治理、动态拓扑、证据质量评估、跨场景泛化和可解释审计增强上。

【建议替换正文】

后续工作将围绕模型能力、协同效率、证据质量和工程化治理继续优化。

第一，进一步完善 Fed-MBPR 与 TruthSeeker Agent 工具链之间的模型服务治理。当前系统已将 Fed-MBPR 底层能力接入 Agent 检测链路，后续可在模型版本管理、服务健康监测、工具降级提示、检测能力标注和跨模态模型评估方面形成更标准的工程机制，使底层模型更新不会破坏上层 Agent 研判流程。

第二，优化多智能体动态拓扑与任务调度策略。当前系统采用阶段式拓扑，适合保证证据闭环和报告可解释性。未来可根据任务类型、输入模态、历史风险和工具可用性动态调整 Agent 顺序、补证次数和质询强度。例如，对纯 URL 风险任务可更早引入 OSINT Agent，对高置信图片生成任务可减少重复质询，对多模态组合样本可提高跨模态一致性检查权重。

第三，强化证据质量评估与自动化复核能力。现有系统已经记录引用覆盖、模型推断占比和残留风险，后续可进一步细化证据质量分级，区分直接工具证据、外部来源证据、用户确认事实、模型综合推断和历史经验引用，并在报告中给出更明确的可采信等级。

第四，提升人机协同效率与经验质量控制。未来可对用户确认的经验进行质量评分、冲突检测和过期提示，避免低质量经验长期影响 Agent 判断。同时，可在协同面板中提供更精细的问题模板和证据推荐，帮助用户更快补充对系统最有价值的信息。

第五，增强公开案例库和个人经验库的大规模检索能力。随着案例和经验持续积累，系统可继续优化向量检索、关键词召回、分类过滤和相似度去重策略，使 RAG 检索结果更稳定、更可解释，并能区分公开案例参考与用户私有经验参考的不同作用。

第六，完善可解释审计和对外报告能力。未来可在现有 Markdown/PDF 报告基础上增加面向不同角色的报告视图，例如技术复核版、管理汇报版和应急处置版；同时强化报告哈希、证据引用、协同摘要和工具降级记录之间的关联，使系统更适合竞赛答辩、真实应急协作和后续责任复核。

通过上述优化，TruthSeeker 将从单次恶意 AIGC 鉴伪工具进一步发展为具备模型持续更新、证据持续沉淀、经验持续复用和报告持续审计能力的跨模态可信研判平台。
