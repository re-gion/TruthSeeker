# TruthSeeker 四 Agent 辩论 + 专家会诊 拓扑详解

## 一、整体拓扑

系统**不是**四个 Agent 同时辩论，而是一条**三阶段流水线**，由 Challenger 把关每个阶段之间的门：

```
START
  │
  ▼
┌──────────┐     ┌────────────┐
│ Forensics │────▶│ Challenger │──┐
└──────────┘     └────────────┘  │
      ▲                │         │ 打回 or 前进
      └────────────────┘         │
                                 ▼
                           ┌──────────┐     ┌────────────┐
                           │   OSINT   │────▶│ Challenger │──┐
                           └──────────┘     └────────────┘  │
                                 ▲                │         │
                                 └────────────────┘         │
                                                           ▼
                                                     ┌───────────┐     ┌────────────┐
                                                     │ Commander  │────▶│ Challenger │──┐
                                                     └───────────┘     └────────────┘  │
                                                           ▲                │         │
                                                           └────────────────┘         │
                                                                                      ▼
                                                                                    [END]
```

**关键设计**：Challenger 是唯一的路由节点。每个阶段完成后都必须经过 Challenger 审查，它决定"放行进入下一阶段"还是"打回当前阶段要求补充分析"。

---

## 二、四个 Agent 各自的职责与工具

| Agent | 角色 | 核心任务 | 可调用工具 |
|-------|------|---------|-----------|
| **Forensics** | 取证分析专家 | 对检材做技术检测，输出结构化取证报告 | Sightengine（图片AIGC检测）、Reality Defender（音视频深伪）、VirusTotal（哈希/URL）、案例RAG检索、Kimi LLM多模态解读 |
| **OSINT** | 开源情报分析师 | 追踪来源、构建溯源图谱、评估威胁 | 文本AI检测+社工风险评分、VirusTotal URL、WhoisXML域名溯源、Exa搜索、案例RAG、溯源图谱构建器、Kimi LLM |
| **Challenger** | 批判性挑战者 | 交叉验证、质询、决定是否放行 | Kimi LLM（输出结构化JSON）、确定性代码检查器（缺失报告/工具失败/低置信度/图谱稀薄等硬规则） |
| **Commander** | 研判指挥官 | 综合所有证据做出最终裁决 | Kimi LLM（最终裁决）、溯源图谱构建器、加权置信度计算器 |

## 三、逐轮实战：案例3（政府通知图片+文本）

假设用户上传了一张"政府通知"图片和一段相关文本。

### 第1轮：Forensics 阶段（取证）

**Forensics Agent 执行：**

1. 用 Kimi 多模态能力**自主观察**图片内容——"这张图片显示的是一份盖有公章的通知文件，排版风格类似某地方政府公告…"
2. 调用 **Sightengine genai** 检测图片是否为 AI 生成 → 返回 87% AI 概率
3. 对文件算 SHA-256 哈希，查 **VirusTotal** → 无已知恶意标记
4. 从文本中提取 URL，查 **VirusTotal URL** → 未发现威胁
5. 搜索**案例 RAG** → 找到类似案例参考
6. 调用 **Kimi LLM** 融合所有结果，输出取证报告：
   - 自主检材观察：图片内容描述
   - 外部检测结果解读：Sightengine 87% AI 生成
   - 融合判断：高概率 AIGC
   - 限制与复核建议

**Challenger 第1次审查（Forensics）：**

- LLM 交叉验证 + 代码硬检查
- 发现问题：`[高] 取证报告未充分讨论 Reality Defender 结果（因样本是图片，该工具不适用，但报告应说明）`
- 置信度：0.72
- **判定：打回** → `requires_more_evidence=True, target_agent="forensics"`

### 第2轮：Forensics 补充分析

**Forensics Agent 再次执行：**

- 新增说明：音视频深伪检测工具不适用于纯图片样本
- 补充 Sightengine 各维度分数细节
- 质量提升

**Challenger 第2次审查（Forensics）：**

- 置信度：0.85
- 质量变化 delta：0.06 < 0.08（收敛）
- 满足三个门控条件：至少2轮 ✓、delta < 0.08 ✓、confidence > 0.8 ✓
- **判定：放行** → 进入 OSINT 阶段

---

### 第3轮：OSINT 阶段（情报分析）

**OSINT Agent 执行：**

1. 对文本做 **AI 文本检测** + **社工风险评分** → 发现典型社工话术模式
2. 提取文本中的 URL，查 **VirusTotal** → 安全
3. 对关联域名做 **WhoisXML 溯源** → 注册时间仅3天，高度可疑
4. **Exa 搜索**脱敏关键词 → 未找到官方来源佐证
5. 构建**溯源图谱**：文本 → URL → 域名 → 注册者（节点+边）
6. Kimi LLM 输出 OSINT 报告：
   - 自主情报推理：域名极新、无官方佐证
   - 外部情报结果：WhoisXML 注册3天
   - 来源可信度：低
   - 关联风险：高度疑似钓鱼/伪造公文

**Challenger 第1次审查（OSINT）：**

- 发现问题：`[中] Exa 搜索查询词过于宽泛，可能遗漏相关结果`
- 置信度：0.78
- **判定：打回** → `target_agent="osint"`

### 第4轮：OSINT 补充分析

**OSINT Agent 再次执行：**

- 优化 Exa 搜索关键词，增加地名+机构名精确搜索
- 补充域名 DNS 历史分析

**Challenger 第2次审查（OSINT）：**

- 置信度：0.88，delta 收敛
- **判定：放行** → 进入 Commander 阶段

---

### 第5轮：Commander 阶段（最终裁决）

**Commander Agent 执行：**

- 汇总 Forensics 取证报告 + OSINT 情报报告 + Challenger 所有历史质询记录
- 加权置信度计算：Forensics 0.85 × 权重 + OSINT 0.88 × 权重
- 输出最终裁决：
  - **结论：高度疑似 AIGC 伪造公文**
  - 置信度：0.87
  - 关键分歧点：无（两个 Agent 结论一致）
  - 后续建议：报告网安部门、封锁域名

**Challenger 第1次审查（Commander）：**

- 置信度：0.90，无高严重度问题
- 满足收敛条件
- **判定：放行** → 流程结束，输出 `final_verdict`

---

## 四、收敛判定的三重门控

Challenger 决定"打回还是放行"不是凭感觉，而是**三个条件同时满足**才放行：

```
放行条件 = 至少2轮 AND 质量变化 < 0.08 AND 置信度 > 0.8
```

| 门控 | 含义 | 阈值 |
|------|------|------|
| 最少轮次 | 每个阶段至少被审查2轮 | ≥ 2 |
| 质量稳定性 | 本轮与上轮质量分差值很小 | delta < 0.08 |
| 置信度 | Challenger 对当前结果有信心 | > 0.8 |

另外有硬上限：**每个阶段最多5轮**。达到5轮仍有高严重度问题时，记录为"残余风险"并强制放行。

## 五、专家会诊（Human-in-the-Loop）

当 Challenger 发现 Agent **卡住了**——连续3轮打回同一个 Agent、都有高严重度问题、置信度始终低于 0.8、质量没有改善——系统判定"自动化辩论无法收敛"，触发**专家会诊**。

### 触发条件（四个必须同时满足）

```
1. 最近3轮都打回同一个 Agent（如连续3次 target_agent="forensics"）
2. 最近3轮都有高严重度问题
3. 当前置信度 < 0.8
4. 相邻轮质量 delta 都 < 0.08（没有改善趋势）
```

### 会诊流程

```
Challenger 检测到卡住
  │
  ▼
LangGraph interrupt() ← 图执行暂停，状态存入 checkpoint
  │
  ▼
创建会诊会话（Supabase）
  │
  ▼
前端收到 SSE: consultation_required
  │
  ├── 用户发起邀请链接（24h 有效）
  │     │
  │     ▼
  │   专家通过链接加入
  │   提交结构化意见（角色/消息/锚定Agent/建议动作/置信度）
  │     │
  │     ▼
  │   用户关闭会话 → Commander 自动生成会诊纪要
  │     │
  │     ▼
  │   用户确认/编辑纪要
  │
  ▼
前端调用 detect/stream?resume=true
  │
  ▼
Command(resume=resume_payload) → 注入专家意见+会诊纪要
  │
  ▼
Commander 节点重新执行（融合专家意见）
  │
  ▼
Challenger 审查 → 放行 → END
```

**核心机制**：用 LangGraph 的 `interrupt()` 暂停图执行，状态自动 checkpoint。恢复时用 `Command(resume=...)` 把人类专家的意见注入到 `expert_messages`，Commander 最多读取10条专家意见纳入最终裁决。

## 六、总结

```
            ┌─────────────────────────────────────────────────┐
            │                 正常自动流程                      │
            │                                                 │
            │  Forensics ──▶ Challenger ──▶ OSINT ──▶ Challenger ──▶ Commander ──▶ Challenger ──▶ END
            │      ▲              │                            │              │
            │      └──────────────┘ 打回                       │              │
            │                     (≤5轮/阶段)                   │              │
            │                                                 │
            └─────────────────────────────────────────────────┘
                              │
                    连续3轮卡在同一Agent
                    置信度<0.8 且无改善趋势
                              │
                              ▼
            ┌─────────────────────────────────────────────────┐
            │               专家会诊流程                        │
            │                                                 │
            │  interrupt() → 人类专家注入意见 → resume()        │
            │  Commander 融合专家意见 → 最终裁决                  │
            │                                                 │
            └─────────────────────────────────────────────────┘
```

简单说：**取证→质询→情报→质询→裁决→质询**，质询者是裁判，它说"不行"就打回重做，说"过了"就前进。如果反复打回都做不好，就暂停请人类专家来帮忙。
