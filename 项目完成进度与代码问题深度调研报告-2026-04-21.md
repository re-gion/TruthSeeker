## 第一部分：项目完成进度调研结果

### 1. 总体结论

TruthSeeker 当前不是纯 UI 壳，也不只是计划文档。当前仓库里已经存在一条真实的比赛演示级主链路：

`上传检材 -> 创建任务 -> detect/stream SSE -> LangGraph 四 Agent -> analysis_states/agent_logs/reports/audit_logs -> 报告下载/分享 -> dashboard 聚合 -> 专家会诊 -> resume`

但它还不是一个完全闭合、可稳定部署、可从零复现数据库、可经受生产安全审计的完整产品。最准确的完成度判断如下：

| 层面 | 当前判断 | 说明 |
| --- | --- | --- |
| 比赛演示核心链路 | 真实可用，约 70%-75% | 上传、任务、SSE、Agent、报告、分享、会诊、大屏都有真实代码闭环。 |
| 工程可交付性 | 部分可用，约 60%-65% | 单测/类型检查能跑一部分，但后端 pytest 和 Next build 在当前环境不可完整通过；部署配置缺失。 |
| 生产可信性 | 部分可用，约 45%-55% | JWT、审计日志、report hash 已有，但 RLS/schema 可复现、持久化 checkpoint、严格错误态、安全边界仍不足。 |
| 白皮书/文档真实性 | 部分一致 | 新文档 `APP_FLOW`、`BACKEND_STRUCTURE` 较准确；`PRD`、`IMPLEMENTATION_PLAN`、白皮书仍有目标架构和当前代码混写。 |

### 3. 前端完成度

#### 真实可用

- 上传入口 `components/upload/FileUploader.tsx` 支持最多 5 个文件、单文件 500MB，前端校验视频/音频/图片/文本类型，先调后端 `/api/v1/upload/`，再调 `/api/v1/tasks` 创建任务。
- 检测页 `app/detect/[taskId]/page.tsx` + `components/detect/DetectConsole.tsx` 会按 `taskId` 拉任务上下文，不再依赖 URL 传 signed file URL。
- `hooks/useAgentStream.ts` 使用 `POST /api/v1/detect/stream` 读取 SSE，处理 `consultation_required`、`consultation_resumed`、`task_failed`、`final_verdict`、`complete`。
- 协作 UI `ExpertPanel.tsx` 会加载历史会诊消息，并向后端 `/consultation/{taskId}/inject` 注入专家意见。
- Dashboard 已接真实后端聚合接口 `/api/v1/dashboard/overview`，并保留外部权威数据来源链接。

#### 部分可用

- 报告页能展示分享报告，但 `simpleMarkdown()` 不支持表格，后端报告中的表格会退化为原始 `|` 文本。
- 检测页的 Markdown 下载是前端本地重新拼接，PDF 下载和分享报告走后端 canonical 报告，两条来源可能不一致。
- 注册页和服务端注册动作存在反馈冲突：页面有“注册成功，请检查邮箱”成功态，但 `auth-actions.ts` 注册成功后直接 `redirect("/")`。

#### 壳子/演示

- `components/demo/DemoCaseSelector.tsx` 的“加载演示”跳到 `/?demo=...`，当前仓库没有找到消费 `demo` 参数的逻辑，案例并未真正装载。
- 案例库真实加载功能在 `task.md` 和 `docs/APP_FLOW.md` 中已明确暂不实现。

### 4. 后端和 API 完成度

#### 真实可用

- `upload.py` 接收文件，检查大小和 MIME，上传 Supabase Storage，返回标准文件对象，并写入 `audit_logs`。
- `tasks.py` 创建任务，保存 `case_prompt`、文件清单、`storage_paths`、模态和优先级，并忽略客户端传入的 `user_id`。
- `detect.py` 解析任务文件，生成 signed URL，启动 LangGraph，流式推送 SSE，写入状态、日志、报告、失败状态和审计日志。
- `report.py` 提供 Markdown/PDF 下载；`share.py` 提供分享 token 生成和公开读取。
- `consultation.py` 支持邀请、专家消息注入、历史消息读取、邀请过期校验。
- `dashboard.py` 从 `tasks`、`reports`、`consultation_invites` 聚合大屏指标。

#### 部分可用

- `SUPABASE_JWT_SECRET=NOT_SET` 时认证中间件不注册，所有受保护路由会失去强制鉴权；这对本地演示方便，但不能按生产安全口径宣称已闭环。
- `tasks.list_tasks` 和 dashboard `_select_rows` 遇到后端异常时返回空列表，容易把 Supabase/权限/网络故障伪装成“没有数据”。
- `detect/stream` 的 HTTP 错误和 SSE 流错误分界仍需继续打磨，特别是长连接启动后的异常可见性。

### 5. 核心业务逻辑、算法和 Agent

#### 真实可用

- `graph.py` 真实定义了 LangGraph 拓扑：START 同时进入 Forensics 和 OSINT，二者完成后进入 Challenger，再路由到 Commander 或打回重审。
- `forensics.py` 调 Reality Defender，失败时降级到 metadata/minimal 结果，并调用 Kimi/Moonshot LLM 解释。
- `osint.py` 处理文本和媒体情报，集成 VirusTotal、元数据和 URL/IOC 提取，失败时降级。
- `challenger.py` 做规则质询、LLM 交叉验证，高冲突时尝试 LangGraph interrupt。
- `commander.py` 汇总证据、权重和专家意见，形成最终裁决。

#### 部分可用/风险

- 会诊恢复依赖 `MemorySaver` 进程内 checkpoint。后端重启后暂停中的任务无法可靠恢复。
- 降级 mock 不完全稳定：`threat_intel.py` 使用 Python `hash(url)`，`deepfake_api.py` 使用随机数，同一输入跨进程可能得到不同分数，不利于审计复现。
- `forensics.py` 里仍有文本检测相关分支，但前面已经对纯文本输入直接返回跳过，后续文本分支实际不可达，属于死代码/契约混乱。
- 白皮书中的 FedPaRS 是研究/训练底座叙事，当前运行时代码没有实现联邦训练、秘密共享、LoRA 聚合、双服务器密文筛查等底层训练链路。

### 6. 数据库、存储和持久化

#### 已实现

- 代码中有写入 `analysis_states`、`agent_logs`、`reports` 和 `audit_logs` 的路径。

### 7. 认证、权限、安全、日志和监控

#### 已实现

- JWT 认证中间件会校验 Supabase JWT，公开路径和受保护路径有区分。
- 上传、任务创建、检测、报告下载、分享创建等写侧接口在认证中间件启用时不是公开路径。

## 第二部分：源代码和文档错误汇报

### P1：严重问题或高优先级风险

#### P1-1 数据库迁移无法从零复现完整 schema/RLS

- 证据：`truthseeker-api/sql/migrations/20260416_full_fix.sql` 只对 `public.tasks` 做 `alter table if exists`，没有创建 `tasks` 表；迁移中未发现 `enable row level security` / `create policy`。
- 矛盾：`task.md` 写“tasks 表已通过 Supabase 迁移创建”和“RLS 策略已配置”。
- 影响：新环境按仓库迁移无法完整建库，部署/评审复现会卡住；权限隔离也无法被版本库证明。
- 建议：补一份基线迁移，包含 `tasks`、核心表、索引、RLS policy，并在测试环境实际执行一次。

#### P1-2 认证中间件依赖 `SUPABASE_JWT_SECRET`，默认 `NOT_SET` 会跳过鉴权

- 证据：`app/main.py` 中 `settings.SUPABASE_JWT_SECRET != "NOT_SET"` 才注册 `AuthMiddleware`；`.env.example` 默认 `SUPABASE_JWT_SECRET=NOT_SET`。
- 影响：如果部署照搬示例值，上传、任务、检测、报告下载、分享创建等受保护接口不会有强制认证。
- 建议：部署前必须配置真实 JWT secret；生产启动时遇到 `NOT_SET` 应直接失败或至少强提示，不应按生产口径放行。

### P2：中高优先级问题

#### P2-1 会诊恢复依赖内存 checkpoint，后端重启后不可恢复

- 证据：`truthseeker-api/app/agents/graph.py` 使用 `MemorySaver`。
- 影响：高冲突任务进入 `waiting_consultation` 后，如果后端进程重启，主持人继续研判可能无法恢复原 LangGraph 状态。
- 建议：改用可持久化 checkpointer，或将 resume 必需状态写入 Supabase 后重建状态。

#### P2-2 前端 Markdown 报告和后端 canonical 报告不一致

- 证据：`DetectConsole.tsx` 下载 MD 时调用前端 `generateMarkdownReport()`；PDF 下载走后端 `/api/v1/report/{taskId}/pdf`；分享页也走后端报告。
- 影响：同一任务可能出现“前端 MD、后端 PDF、分享页”三种内容不一致，且前端 MD 不参与后端 `report_hash`。
- 建议：前端 MD 下载改为调用 `/api/v1/report/{taskId}/md`，或明确标注为本地预览稿。

#### P2-3 后端故障会被部分接口伪装为空数据

- 证据：`tasks.py:list_tasks` 捕获异常后返回 `[]`；`dashboard.py:_select_rows` 出错也返回空列表。
- 影响：Supabase 失败、权限错误、网络异常会被用户误解为“没有任务/没有指标”。
- 建议：返回可识别的错误态或 `dataWarnings`，前端明确展示“数据源异常”。

#### P2-4 文本文件上传校验偏弱

- 证据：`upload.py` 允许 `text/plain`，但对真实内容是否可解码、文本比例、二进制伪装没有更强检查。
- 影响：二进制内容伪装成文本可能进入 OSINT 文本通道。
- 建议：至少增加 UTF-8/常见编码可解码检查、控制字符比例检查、文件扩展名和 MIME 双重校验。

#### P2-5 降级 mock 不稳定，影响审计复现

- 证据：`threat_intel.py` 使用 Python `hash(url)`，`deepfake_api.py` 使用随机数。
- 影响：同一检材在不同进程或重试中可能得到不同分数，报告 hash 和排障复现都会受影响。
- 建议：用 SHA-256 等稳定输入派生值替代 Python hash/random。

#### P2-6 前端演示案例入口没有真实消费链路

- 证据：`DemoCaseSelector.tsx` 跳转 `/?demo=...`，未找到读取 `demo` 参数的代码。
- 影响：用户点击“加载演示”后看似有动作，但案例没有真正载入检测台。
- 建议：要么接入真实案例状态，要么在 UI 中明确标为展示卡片。

#### P2-7 报告分享页 Markdown 渲染能力不足

- 证据：`app/report/[taskId]/page.tsx` 的 `simpleMarkdown()` 只处理标题、粗体、代码和列表。
- 影响：后端报告中的表格会显示为原始 Markdown，降低报告可读性。
- 建议：支持表格，或使用受控 Markdown 渲染器。

#### P2-8 后端业务测试不能在当前机器完整运行

- 证据：`python -m pytest tests`、`test_report_and_consultation_api.py`、`test_dashboard_api.py` 均被 `WinError 10106` 阻塞。
- 影响：报告、会诊、dashboard API 的集成测试无法在本机完成验证。
- 建议：固定 Python 3.11/3.12 环境，禁用不必要 pytest 插件，修复 Windows asyncio/_overlapped 初始化问题。

#### P2-9 Next 生产构建不可证明通过

- 证据：`node node_modules/next/dist/bin/next build` 304 秒超时，诊断停在 `buildStage: "compile"`。
- 影响：当前不能证明前端生产构建可复现，部署风险未解除。
- 建议：在干净 Node 环境和稳定网络下重跑 build，必要时排查 Next build worker、字体、R3F/Three 依赖和 `.next` 状态。

### P3：普通问题、文档偏差或维护风险

#### P3-1 注册成功态和实际跳转冲突
