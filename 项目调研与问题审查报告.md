## 一、项目完成进度调研结果

### 1. 总体完成度判断

当前项目不是空壳，已经具备较完整的比赛演示主链路：

`上传/建任务 -> FastAPI 检测流 -> LangGraph 四智能体分析 -> SSE 前端展示 -> 结果持久化 -> 报告/PDF/分享 -> 数据大屏汇总`

综合判断：

- **比赛演示完成度**：约 **75% - 80%**。核心 UI、检测流、报告和大屏已经能讲出完整故事，且多处已经接真实后端接口。
- **工程化/产品化完成度**：约 **60% - 65%**。主要短板在测试可复现性、部署构建稳定性、权限边界、会诊历史闭环、案例库真实加载、文档失真，以及 FedPaRS 底层算法与当前运行时代码之间的落差。
- **最关键结论**：项目已经从“纯展示页”推进到“核心链路可运行的比赛系统”，但还不能把白皮书中“FedPaRS/PARSIFAL 底层训练基座、审计日志、防篡改报告、pgvector 长期记忆”等内容说成当前仓库已经落地。

### 2. 前端完成情况

#### 已完成内容

- **上传与任务创建是真接后端**：`FileUploader.tsx` 会调用 `/api/v1/upload/` 上传文件，再调用 `/api/v1/tasks` 创建任务，并把任务 ID 跳转到检测页。
- **检测控制台是真接 SSE**：`useAgentStream.ts` 通过 `POST /api/v1/detect/stream` 消费后端 SSE，能处理 `agent_log`、`forensics_result`、`osint_result`、`challenger_feedback`、`timeline_update`、`final_verdict`、`complete` 等事件。
- **报告/PDF/分享有后端闭环**：检测完成后前端可下载 Markdown/PDF，也可调用 `/api/v1/share/{taskId}` 生成分享链接；分享页会按 token 请求 `/api/v1/share/{token}`。
- **数据大屏已从纯硬编码推进到真实聚合接口**：`lib/dashboard/index.ts` 请求 `/api/v1/dashboard/overview`，再做字段归一化；大屏仍保留权威外部数据模块，但系统能力层来自后端聚合。
- **前端测试基础已补齐**：已有 Vitest 配置和 dashboard 相关测试，当前直接 Node 入口可跑通。

#### 半完成 / 伪完成内容

- **文本提示词没有进入检测流**：上传页把 `prompt` 放进 URL，但 `DetectConsole.tsx` 没有读取这个参数，后端 `DetectRequest` 也没有 `prompt` 字段。结果是用户输入的文本提示只进入任务描述，没有真正进入四智能体检测状态。
- **案例库仍是演示壳**：`DemoCaseSelector.tsx` 只是跳转到 `/?demo=...`，当前没有发现下游页面真正消费 `demo` 参数并加载案例素材或预置状态。
- **专家会诊前端缺历史回放**：`ExpertPanel.tsx` 通过 Supabase broadcast 做实时消息，并调用后端注入接口，但没有读取 `/messages` 或 `/unread`，刷新或晚加入时看不到历史会诊记录。
- **分享页路由命名容易误导**：`app/report/[taskId]/page.tsx` 实际把 `taskId` 当成 `share token` 使用，功能能跑，但维护语义不清。
- **Landing 页 FedPaRS 展示偏“研究叙事”**：`FederatedLearning.tsx` 展示 FedPaRS CORE、节点聚合和 40% 拜占庭攻击鲁棒等内容，但当前运行时代码没有对应的联邦训练实现。

#### 判断依据

- 前端接口调用位置：`truthseeker-web/components/upload/FileUploader.tsx`、`truthseeker-web/hooks/useAgentStream.ts`、`truthseeker-web/components/detect/DetectConsole.tsx`。
- 大屏数据层：`truthseeker-web/lib/dashboard/index.ts`、`truthseeker-api/app/api/v1/dashboard.py`。
- 演示案例：`truthseeker-web/components/demo/DemoCaseSelector.tsx`。
- 会诊面板：`truthseeker-web/components/collaboration/ExpertPanel.tsx`、`truthseeker-web/hooks/useRealtimeSession.ts`。

### 3. 后端完成情况

#### 已完成内容

- **FastAPI 路由主干已完整注册**：`detect`、`tasks`、`upload`、`report`、`consultation`、`share`、`dashboard` 均已挂载到 `/api/v1`。
- **四智能体 LangGraph 不是占位**：`graph.py` 构建了 `Forensics -> OSINT -> Challenger -> Commander` 的状态图，并支持 Challenger 条件打回 Forensics/OSINT。
- **外部能力与降级存在真实逻辑**：Reality Defender、VirusTotal、Kimi/LLM 客户端和降级控制器都有实现；外部服务失败时会降级到元数据/规则/最小结果，而不是直接崩溃。
- **检测结果持久化已有明显推进**：`detect.py` 调用 `AnalysisPersistenceService`，把中间状态写入 `agent_logs`、`analysis_states`，最终裁决 upsert 到 `reports`，并更新 `tasks.result`。
- **报告与分享接口可形成闭环**：`report_generator.py` 可从 `tasks + analysis_states + agent_logs + reports` 组装 Markdown/PDF；`share.py` 可生成 token 并公开访问分享报告。
- **数据大屏后端聚合已实现**：`dashboard.py` 汇总 `tasks`、`reports`、`consultation_invites`，并过滤敏感字段。

#### 半完成 / 风险点

- **检测失败不会可靠落库为 failed**：`detect.py` 异常时会发送 `error` SSE，但只有拿到 `final_verdict_data` 才更新任务完成状态。若图执行异常，任务可能长期停在 `analyzing`。
- **身份信任边界偏宽**：`tasks.py`、`upload.py`、`detect.py` 仍允许客户端传入的 `user_id` 影响任务归属、存储路径或状态。公开路由上这有冒名风险。
- **认证是“可选启用”**：如果 `SUPABASE_JWT_SECRET=NOT_SET`，认证中间件不会注册；这适合演示，但不能作为生产安全状态。
- **会诊邀请没有真实过期**：邀请过期时间写死到 2999 年，校验逻辑主要看 `status`，不是完整生命周期控制。
- **限流是内存级实现**：`RateLimitMiddleware` 基于进程内字典，重启丢失，多实例部署不共享。

#### 判断依据

- 路由：`truthseeker-api/app/api/v1/router.py`。
- 检测流与持久化：`truthseeker-api/app/api/v1/detect.py`、`truthseeker-api/app/services/analysis_persistence.py`。
- 报告分享：`truthseeker-api/app/services/report_generator.py`、`truthseeker-api/app/api/v1/report.py`、`truthseeker-api/app/api/v1/share.py`。
- 会诊：`truthseeker-api/app/api/v1/consultation.py`、`truthseeker-api/app/agents/nodes/challenger.py`。
- 安全中间件：`truthseeker-api/app/middleware/auth.py`、`rate_limit.py`。

### 4. 接口与数据流完成情况

#### 已打通链路

- 前端上传文件 -> 后端 Supabase Storage -> 返回 signed URL。
- 前端创建任务 -> 后端写 `tasks`。
- 检测页启动 SSE -> 后端 LangGraph 流式输出 -> 前端实时展示 Agent 状态与日志。
- 检测完成 -> 后端写 `tasks.result`、`analysis_states`、`agent_logs`、`reports`。
- 报告下载 -> 后端读取分析表组装 Markdown/PDF。
- 分享报告 -> 后端生成 share token -> 前端用 token 访问公开报告。
- Dashboard -> 前端调用后端聚合接口 -> 展示真实任务/报告/会诊统计。

#### 尚未真正闭环链路

- **文本 prompt -> 检测 Agent**：当前断在前端检测页/后端请求模型之间。
- **案例库选择 -> 加载真实案例 -> 检测流**：当前基本只是 UI 卡片与跳转。
- **会诊历史 -> 新加入专家/刷新后恢复上下文**：后端有消息接口，前端未消费。
- **失败检测 -> failed 状态 -> 前端/大屏正确体现失败**：后端失败落库不完整。
- **数据库迁移 -> 远端真实库完整 schema**：仓库里迁移文件仍不包含 RLS 策略和部分声明的高级表；本次只补了缺失的会诊消息表迁移。

### 5. 算法与核心能力完成情况

#### 已实现

- **四智能体推理编排已实现**：Forensics、OSINT、Challenger、Commander 都有实际节点逻辑。
- **收敛/打回机制有代码**：Challenger 可根据证据不足或矛盾打回 Forensics/OSINT；最大轮数兜底存在。
- **外部检测与情报 API 有集成封装**：Reality Defender、VirusTotal、LLM、文本检测、元数据提取等均有对应工具层。
- **降级机制已实现**：当外部服务不可用时，系统会输出降级结果并记录降级状态。

#### 未实现或不能按白皮书口径宣称已实现

- **FedPaRS 不是当前运行时算法代码**：根目录 `FedPaRS核心算法.md` 描述的是 FedLLM/LoRA/秘密共享/FWHT/SRHT/密文检测/鲁棒聚合方案，但仓库运行时没有对应训练代码、客户端参与方、双服务器秘密共享协议、LoRA 更新聚合实现。
- **PARSIFAL/FedPaRS 当前更像“研究基座/算法方案”**：当前实际检测依赖 Reality Defender、VirusTotal、LLM 和启发式降级，不能写成“底层 FedPaRS 已训练出专精模型并被系统实时调用”。
- **白皮书中的审计日志、报告 hash、pgvector 记忆库未落地**：代码中未发现 `audit_logs`、`report_hash`、`pgvector`、`threat_vectors`、`agent_memories` 等运行时实现。

## 二、代码与文档问题汇报

### 1. 已顺手修复的小问题

#### 1.1 补齐会诊消息表迁移

- **位置**：`truthseeker-api/sql/migrations/20260420_consultation_messages.sql`
- **问题**：代码已经读写 `consultation_messages`，但当前仓库迁移里没有创建该表。
- **修改内容**：新增迁移，创建 `public.consultation_messages`，包含 `task_id`、`role`、`message`、`expert_name`、`created_at`，并增加 `task_id + created_at` 索引。
- **修改原因**：这是明确的代码/数据库 schema 不一致，会导致专家会诊消息注入和 Challenger 读取专家意见失败。
- **验证**：`python -m compileall app tests` 通过；SQL 迁移未连接真实 Supabase 执行，需在部署库上应用迁移。

### 2. 发现但未修改的大问题 / 严重问题

#### 2.1 文本提示词未进入检测流

- **位置**：`FileUploader.tsx`、`DetectConsole.tsx`、`detect.py`
- **性质**：前后端数据流断裂。
- **表现**：上传页把 prompt 放到 URL 参数，但检测页不读，后端请求模型也没有 prompt 字段。纯文本分析任务最后只传 `mock://text-input`。
- **影响**：用户输入的案件描述、提示词和文本证据不会真正参与检测，文本通道的实际业务能力被高估。
- **暂不直接修改原因**：需要设计后端 State 字段、Forensics/Text Detection 输入来源、报告展示和数据库记录，影响多模块。
- **建议**：给 `DetectRequest` 增加 `text_content`/`prompt` 字段；检测页读取并传入；后端写入 `input_files` 或独立 `text_payload`；文本检测工具优先使用真实文本而非 mock URL。

#### 2.2 检测失败不会可靠标记任务失败

- **位置**：`truthseeker-api/app/api/v1/detect.py`
- **性质**：任务状态闭环缺失。
- **表现**：异常时发 `error` SSE，但没有统一把 `tasks.status` 更新为 `failed`，也不一定写 `completed_at/updated_at`。
- **影响**：任务、大屏、报告生成和用户界面可能长期看到 `analyzing`。
- **暂不直接修改原因**：需要确认前端对失败态的预期、是否需要保留部分分析结果、失败事件是否也要写报告/日志。
- **建议**：引入 `mark_task_failed(task_id, error_summary)`，异常路径落库；前端消费 error 后显示失败状态和重试入口。

#### 2.3 用户身份和任务归属信任过宽

- **位置**：`tasks.py`、`upload.py`、`detect.py`、`auth.py`
- **性质**：权限与安全风险。
- **表现**：公开路由允许请求体或表单里的 `user_id` 影响任务/存储归属；JWT secret 未配置时认证整体降级。
- **影响**：存在冒名、数据归属混乱和路径隔离失效风险。
- **暂不直接修改原因**：会影响匿名演示、登录用户、RLS 策略和现有测试，需要明确比赛演示与生产安全的取舍。
- **建议**：认证用户只信 `request.state.user_id`；匿名用户统一 `anonymous/anon`；禁止客户端传入 user_id 覆盖认证身份；部署环境必须配置 `SUPABASE_JWT_SECRET`。

#### 2.4 FedPaRS 运行时代码缺失

- **位置**：`FedPaRS核心算法.md`、白皮书、Landing 页 FedPaRS 组件。
- **性质**：研究算法与运行代码脱节。
- **表现**：算法文档描述 LoRA、秘密共享、SRHT/FWHT、密文检测、鲁棒聚合，但当前后端没有对应训练/聚合执行代码。
- **影响**：如果答辩或白皮书宣称“系统已实现 FedPaRS 底层训练基座”，容易被追问时穿帮。
- **暂不直接修改原因**：这不是小改能补的功能，需要算法工程实现、实验数据和接口封装。
- **建议**：短期把 FedPaRS 定位为“底层研究方案/未来可替换模型训练基座”；当前系统定位为“多智能体检测编排 + 外部检测 API + 降级策略”。

#### 2.5 生产构建和后端测试不可复现

- **位置**：前端构建、后端 Python 环境。
- **性质**：部署与交付风险。
- **表现**：Next build 当前卡在 compile；后端 pytest 缺依赖；Python 导入 `asyncio/supabase` 触发 WinError 10106；旧 venv 指向不存在的 Python。
- **影响**：项目无法证明“一键构建/测试/部署可用”。
- **暂不直接修改原因**：涉及本机 Python/Windows 网络栈、依赖重装和可能的环境迁移，不属于小范围代码修复。
- **建议**：重建干净 Python 3.11/3.12 venv；按 requirements 安装；补一个 `scripts/check-env`；前端清理 `.next` 后用稳定 Node 环境重跑 build。

`FedPaRS核心算法.md` 已经明确当前技术方案是 FedPaRS，且它面向多模态 FedLLM 参数高效微调，加入了 LoRA、模态分离、FWHT/SRHT、分块量化、混合秘密共享、密文筛查、缩放因子软截断和鲁棒聚合。这已经不是简单复述 PARSIFAL。

继续写 PARSIFAL 会带来两个误导：

- 误导评审以为项目只采用原始 PARSIFAL，而不是结合本项目多模态/LLM/警务研判需求演进。
- 误导评审以为当前代码已经完整实现 PARSIFAL/FedPaRS 训练闭环。

#### 4.3 FedPaRS 与 PARSIFAL 的关系

建议表述为：

> FedPaRS 是在 PARSIFAL 的“隐私保护 + 鲁棒聚合 + 草图检测”思想基础上，结合 TruthSeeker 多模态 Deepfake 鉴伪和 FedLLM 参数高效微调需求演进出的技术方案。它保留了秘密共享和鲁棒筛查的安全目标，同时引入 LoRA 参数更新、视觉/文本模态分离、结构化旋转、低比特压缩、缩放因子异常检测和面向多模态证据的鲁棒更新流程。

#### 4.4 应如何更新为符合当前项目实际的表述

建议按章节处理：

- **第 2 节核心架构**：把“底层感知基座采用 PARSIFAL”改为“底层研究基座采用 FedPaRS 方案，当前工程实现先通过第三方检测 API 与多智能体编排承接检测能力”。

#### 4.5 是否已直接修改白皮书

## 五、最终结论

TruthSeeker 当前已经具备较强的比赛演示形态：核心检测控制台、四智能体后端、SSE、报告、分享、大屏都不再是纯静态页面。但它距离“完整可信产品”仍有明显差距，尤其是文本输入链路、失败状态闭环、权限归属、会诊历史、案例真实加载、后端测试/部署环境和白皮书技术叙事。

最需要马上统一的是白皮书口径：**PARSIFAL 应更新为 FedPaRS，但 FedPaRS 也不能被写成当前仓库已经完整实现的运行时代码。**更准确的说法是：FedPaRS 是基于 PARSIFAL 思想演进出的本项目底层研究方案，而当前工程实现主要完成了上层多智能体研判、外部检测 API 接入、降级策略、报告与数据展示闭环。
